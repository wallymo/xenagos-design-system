Principles

Every screen should read as confident, scientifically sophisticated, and investor-ready. These five rules resolve most design decisions before they become questions.

01 — One accent

Blue is the only accent.

Every interactive element, highlight, and second headline line uses the blue ramp. No teal, no green, no secondary palette. Restraint is the brand.

02 — Light / dark rhythm

Sections alternate, full-bleed.

Pages breathe between white/gray-50 bands and navy bands. Dark bands carry the molecular imagery; light bands carry density and detail.

03 — Two-tone headlines

Line one ink, line two blue.

The signature typographic move, finished with a 48×2px accent rule. Light weight at large sizes — never bold display type.

04 — Motion is atmosphere

Nothing flashy, ever.

Fades, soft parallax, progressive reveals, slow helix drift. Motion reinforces sophistication without drawing attention to itself. Honor reduced-motion.

05 — Components, not pages

Every element here ships as a reusable component with content passed via props. Pages are compositions. If a layout can't be expressed by the components on this page, the system grows first — the page doesn't go one-off. ⚠ hex & type sizes pending brand-asset confirmation

Motion

Three durations, one easing curve, four patterns. Motion is atmosphere — fades, soft parallax, progressive reveals, slow helix drift. Hover the cards to compare durations.

fast · 200ms
hovers, arrows, color shifts
base · 400ms
card lifts, panel transitions
slow · 700ms
scroll-in reveals, hero entrances
Rules
PatternSpec
easingcubic-bezier(0.16, 1, 0.3, 1) — ease-out, entrances only; never bounce or elastic
revealfade + 16px rise on scroll-in, staggered 80–120ms within a band (this page demonstrates it)
parallaxsoft, on helix/molecular backgrounds only — never on text
driftcontinuous slow helix float, ±12px over ~14s
reduced-motionprefers-reduced-motion disables drift, parallax, and reveals — content shows immediately

Stack & Structure

This is the developer half of the system. The website is built as a scalable design system, not a collection of pages — every decision serves reusability, modularity, consistency, maintainability, and future expansion. Conventions below mirror the live repo HedgeHox/xenagos-web.

Tech stack — required & in use
LayerToolNotes
frameworkReact + ViteSPA prerendered to static HTML at build
languageTypeScriptstrict: true · noUnusedLocals/Parameters · path alias @/* → src/*
lintingESLinttypescript-eslint + react-hooks — must pass clean
stylingSass / SCSStokens + mixins, BEM class naming
routingReact Routerstatic routes, lazy per page
prerendervite-react-ssgevery route emits crawlable static HTML
motionFramer Motionstaggered reveals & soft transitions only
Dependency philosophy: lean and intentional. Additional libraries are added only for animation, motion, transitions, scrolling, or performance — never for convenience.

The folder rule

Every reusable UI element lives in its own folder under components/, named in kebab-case, holding a paired component + style file:

components/feature-card/

feature-card.tsx  +  feature-card.scss

There are 25 components on this pattern today. Two exceptions ship behaviour, not visuals, so they carry no .scss: seo-head (meta tags) and reveal-group (scroll logic).
15

Component Architecture

All UI is componentized. Each component splits cleanly into a .tsx for logic and markup and a .scss for its own styling — and pages are compositions of component calls fed by props, never hardcoded layouts.

name.tsx  is responsible for
  • collecting props
  • conditional rendering
  • rendering markup
  • handling state / logic
  • exposing configurable properties
name.scss  is responsible for
  • component-specific styling only
  • responsive styling
  • modifier states
  • animations / transitions where applicable
  • nothing global — tokens come from styles/
Pages are component calls with content via props
pages/Home.tsx — composition, not markup// copy lives in content/home.ts, imported as `c`
<SeoHead title={c.seo.title} description={c.seo.description} path="/" />

<Hero {...c.hero} />

<SectionBand tone="dark" texture>
  <SectionIntro {...c.approach} onDark />
  {c.approach.features.map((f, i) => (
    <FeatureCard key={f.title} {...f} icon={stepIcons[i]} />
  ))}
</SectionBand>
Avoid
✕hardcoded page content inside pages  ·  duplicated section markup  ·  inline styling  ·  one-off components  ·  page-specific logic where a reusable solution is possible
Why it matters: content and layout must be able to evolve quickly — new pages, sections, leaders, and stories — without structural rewrites. If a layout can't be expressed by existing components, the system grows first; the page never goes one-off.

Consistency is enforced through naming. One casing rule per layer, applied everywhere — so any developer can predict a file, class, component, or token name without looking it up.
